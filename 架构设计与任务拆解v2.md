## 一、总体架构

采用 **分层模块化** 架构，Flutter 跨平台 UI，核心逻辑用纯 Dart 实现。**发送和接收各有独立的 Engine Isolate**，与 UI 线程完全隔离，平台差异通过薄通道层抽象。

```
┌──────────────────────────────────────────────────────┐
│  UI Isolate (Flutter Main)                           │
│  ┌──────────────────────────────────────────────────┐│
│  │ Presentation (Widgets)                          ││
│  │ 设备与发现 | 传输 | 历史 | 设置                  ││
│  ├──────────────────────────────────────────────────┤│
│  │ State Management (Riverpod, 6 providers)        ││
│  └──────────────────────────────────────────────────┘│
├──────────────────────────────────────────────────────┤
│  Engine Isolates (Dart Isolate × 2)                  │
│  ┌────────────────────┐ ┌─────────────────────┐     │
│  │ TransferEngine     │ │ ReceiveEngine       │     │
│  │ (发送端)           │ │ (接收端)            │     │
│  │ FileReader / CRC32 │ │ FrameParser / Write │     │
│  │ SocketWriter       │ │ ACK / Resume        │     │
│  │ TokenBucket        │ │                     │     │
│  └────────────────────┘ └─────────────────────┘     │
├──────────────────────────────────────────────────────┤
│  Business Logic (Pure Dart, runs in UI Isolate)      │
│  ┌──────────┐ ┌───────────┐ ┌──────────────────┐    │
│  │ Discovery│ │ Connection│ │ Clipboard/History│    │
│  │ UDP 广播  │ │ Manager   │ │ /Settings        │    │
│  └──────────┘ └───────────┘ └──────────────────┘    │
├──────────────────────────────────────────────────────┤
│  Network Transport (Dart:io + custom FLP protocol)   │
│  UDP Broadcast Discovery | TCP Server/Client        │
├──────────────────────────────────────────────────────┤
│  Platform Abstraction Layer (MethodChannel)           │
│  Network interfaces | WiFi lock | File access |      │
│  Drag & Drop | Clipboard | Permissions               │
├──────────────────────────────────────────────────────┤
│  Host Platform (Android 8+, Windows 10 1809+)        │
└──────────────────────────────────────────────────────┘
```

**关键设计原则**：

- UI 与逻辑完全解耦，业务模块通过 Riverpod 与 UI 通信
- **发送和接收分别运行在独立的 Engine Isolate 中**：`TransferEngine` 负责文件读取、分片、CRC32 校验、Socket 写入；`ReceiveEngine` 负责帧解析、磁盘写入、ACK 生成
- UI Isolate 仅接收进度事件，不直接参与文件 I/O 或协议解析
- 平台通道仅暴露最小必要功能，业务不依赖平台

## 二、模块职责

### 2.1 设备发现模块 (DiscoveryService)

- 基于 **UDP 广播**：向 `255.255.255.255:45679` 及子网定向广播地址周期性发送 `fastshare_hello` JSON 数据报
- 维护在线设备列表（去重、心跳 15s 超时、离线移除）
- 多网卡绑定：根据用户选择或自动策略绑定特定接口
- Android 端通过 WiFi 多播锁 (`WifiManager.MulticastLock`) 保持后台可发现
- 输出设备上线/下线事件流

### 2.2 连接与配对管理 (ConnectionManager)

- 信任设备存储（SQLite，`TrustedDeviceRepository`）
- TCP 长连接池：按需建立，空闲回收，支持重连
- 配对流程 (PairingProtocol)：6 位随机码挑战，3 次握手 (PAIR_REQUEST → PAIR_CONFIRM → PAIR_RESULT)，确认后持久化 SHA256 Token
- 连接认证：信任设备免确认，非信任走配对弹窗
- 接收端传输 Offer 管理：`acceptPendingOffer()` / `rejectPendingOffer()`

### 2.3 文件传输引擎

**发送端 — TransferEngine**（独立 Dart Isolate）：

- 接收 UI 传入的预扫描文件列表（通过 `file_picker` 或系统分享获取，非引擎内扫描）
- **队列管理**：任务排队、暂停/恢复、取消
- **传输模式判定器**：根据文件大小自动选择顺序/并发/混合模式
- **动态并发调整器**：监测吞吐量、IO、内存，自动微调
- **块传输控制器**：1MB 分块、CRC32 校验、ACK 确认、断点续传
- **文件夹处理**：序列化目录结构，先建树后传文件
- **速率控制**：令牌桶算法，支持手动限速
- **进度汇报**：经去抖处理，以不超过 10 次/秒的频率向 UI Isolate 发送进度事件
- **资源释放**：传输完成或取消后 Engine Isolate 自动销毁，不保留空转 Isolate

**接收端 — ReceiveEngine**（独立 Dart Isolate）：

- 从 TCP Socket 流式读取 FLP Frame，解析帧头/负载
- 控制帧 (OFFER/CANCEL/PAUSE/RESUME) → 中继到 UI Isolate 处理
- 数据帧 (CHUNK) → CRC32 校验，写入磁盘，回复 ACK
- 错误处理：CRC 不匹配发送 NACK，磁盘空间不足通知 UI，异常时发送 `transfer_complete(success: false)` 通知发送端

### 2.4 剪贴板共享模块

- 通过现有 TCP 连接推送文本，接收端通过 `Clipboard.setData()` 写入系统剪贴板

### 2.5 历史记录与设置

- 历史记录：SQLite 存储（`HistoryRepository`），可查询、清理
- 用户设置：信任列表、并发、限速、网卡、深色模式、保护阈值、下载路径等（SharedPreferences + SQLite）

### 2.6 状态管理 (Riverpod)

主要 Provider（共 6 个）：

| Provider | 文件 | 职责 |
|---|---|---|
| `onlineDevicesProvider` | `discovery_provider.dart` | 在线设备列表 + 上下线事件流 |
| `connectionStateProvider` | `connection_provider.dart` | 连接状态、TransferOffer 接收、配对流程、ReceiveEngine 管理 |
| `activeTransferProvider` | `transfer_provider.dart` | 当前活跃的发送任务 + TransferEngine Isolate 生命周期 |
| `transferQueueProvider` | `transfer_provider.dart` | 等待中的传输任务队列 |
| `receiveTransferProvider` | `connection_provider.dart` | 当前活跃的接收任务 |
| `settingsProvider` 系列 | `settings_provider.dart` | 用户配置（名称/端口/限速/深色模式/下载路径等） |

另有 `navigationProvider` 管理底部导航 Tab 索引，`clipboardProvider` 管理剪贴板推送/接收。

### 2.7 平台通道 (PlatformChannel)

抽象接口 `PlatformInterface`，Android 通过 MethodChannel 实现：

- `getNetworkInterfaces()` → IP、网卡名、类型
- `acquireMulticastLock()` / `releaseMulticastLock()` → Android WiFi 锁
- `getSharedFiles()` → Android 系统分享接收
- `copyToClipboard(text)` `getClipboardText()`
- `getDeviceModel()` → Android `Build.MODEL`
- Windows 端通过 FFI 和 Win32 API 实现拖拽支持 (`desktop_drop`)

## 三、Engine Isolate 详细设计

### 3.1 设计动机

Dart 单线程事件循环中，文件分片、CRC32 校验、大量磁盘 I/O 等 CPU 密集操作即使使用异步 Future，仍会抢占 UI 线程时间导致卡顿。将传输引擎独立到专用 Isolate，利用多核并保证 UI 线程绝对流畅。

### 3.2 发送端消息通信模型 (TransferEngine)

```
UI Isolate                          TransferEngine Isolate
   │                                    │
   │ ──── start_transfer ────────────►  │  (携带预扫描的文件列表)
   │                                    ├─ 判定传输模式
   │ ◄─── progress (≤10/s) ──────────── │
   │ ◄─── speed ─────────────────────── │
   │ ◄─── file_complete ──────────────  │
   │ ◄─── error ──────────────────────  │
   │                                    │
   │ ──── pause / resume / cancel ───►  │
   │                                    │
   │ ◄─── transfer_complete ──────────  │
   │                                    ├─ 销毁 Isolate
```

### 3.3 接收端消息通信模型 (ReceiveEngine)

```
UI Isolate                          ReceiveEngine Isolate
   │                                    │
   │ (TCP Socket 已由 UI Isolate 建立)  │
   │ ──── start_receive(socket) ────►   │
   │                                    ├─ 流式读取 FLP Frame
   │ ◄─── transfer_offer ─────────────  │  (中继到 UI 显示确认弹窗)
   │ ──── accept / reject ──────────►   │
   │                                    ├─ 接收数据块, CRC32 校验
   │ ◄─── progress ──────────────────── │
   │ ◄─── file_complete ──────────────  │
   │ ◄─── transfer_complete ──────────  │
   │                                    ├─ 销毁 Isolate
```

### 3.4 命令与事件定义

在 `commands.dart` 中定义。UI → Engine 发送 `EngineCommand`，Engine → UI 发送 `EngineEvent`。命令类型：`start_transfer`、`pause`、`resume`、`cancel`、`shutdown`。事件类型：`progress`、`speed`、`file_complete`、`mode_change`、`error`、`transfer_complete`。

### 3.5 进度事件去抖

Engine Isolate 内部每 1 秒采样一次性能数据，但向 UI 发送的进度更新限制为 **每 250ms 最多一次**，避免过度刷新。

### 3.6 特别注意事项

- **Isolate 生命周期**：传输完成/取消后立即 `kill()` 释放资源；无传输任务时不保留空转 Isolate
- **文件列表**：由 UI Isolate 通过 `file_picker` 或系统分享 Intent 收集，以 `FileTransferItem` 列表形式传入 Engine。Engine 不负责文件系统遍历
- **文件权限**：Android content URI 等需在主 Isolate 获取读权限，将路径传给 Engine，Engine 只负责读取文件内容
- **错误隔离**：Engine Isolate 内部未捕获异常不会导致 UI 崩溃，会封装为 `error` 事件发送给 UI
- **旧版接收器**：`lib/business/transfer/file_receiver.dart` 已标记为 `@Deprecated`，接收逻辑已迁移至 `ReceiveEngine`

## 五、通信协议概要（遵循 FLP v1.2）

- **服务发现**：UDP 广播到 `255.255.255.255:45679`，数据报为 JSON `fastshare_hello`，携带设备 ID、名称、端口、平台、版本
- **传输端口**：TCP 默认 **34568**（可配置，旧版 45678 自动迁移）
- **消息格式**：
  - 控制消息：UTF-8 JSON，封装在统一 FLP Frame 中
  - 数据块：二进制结构体，包含 chunk 索引、偏移、数据及 CRC32 校验
- **关键特性**：版本协商、ACK 确认、断点续传、剪贴板推送
- **强制约束**：UI Isolate 不得直接接触 Socket 及协议解析，所有 FLP 操作必须在 Engine Isolate 完成
